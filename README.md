# Harness Engineering for AI Coding Assistants

A practical way to make AI-assisted development more reliable, reviewable, and controlled.

> AI can generate code quickly. The hard part is creating a workflow that keeps the output safe, consistent, and maintainable.

> Guides: English `docs/USER_GUIDE.md` | Traditional Chinese `docs/USER_GUIDE_zh-TW.md`

## Overview

This repository packages a **Harness Engineering workflow for AI coding assistants** (Cursor and Kiro).

Instead of relying on prompts alone, it gives AI agents a working environment with:

- explicit repo rules
- path-based risk control
- short-term and long-term memory
- reusable skills and specialized subagents
- example CI guardrails

The goal is simple: help AI agents work inside a defined system, not in an unbounded chat.

## IDE Support

| IDE | Implementation | Status |
|-----|----------------|--------|
| **Cursor** | `.cursor/` (compiled) | ✅ Full support |
| **Kiro** | `.kiro/` (compiled) | ✅ Full support |

**Source-based workflow**: All rules, agents, skills, and hooks are maintained in `src/` and compiled to IDE-specific formats using `tools/harness-compiler.js`.

Both implementations share:
- `AGENTS.md` (project contract)
- `risk-tiers.json` (risk classification)
- `docs/` (memory system)
- `examples/` (PHP guardrails, CI workflows)
- `src/` (unified source for rules, agents, skills, hooks)

## Why this exists

AI coding assistants are powerful, but raw agent autonomy can drift quickly:

- architecture gets inconsistent
- risky paths are changed too casually
- context is lost between sessions
- review standards become unclear

This repo is an attempt to solve that by turning the repo itself into a control surface.

## Kiro Implementation

### Steering Files (`.kiro/steering/`)

| File | Inclusion | Role |
|------|-----------|------|
| `core-protocol.md` | `always` | Task start order, delegation map |
| `php-guardrails.md` | `fileMatch: **/*.php` | PHP guardrail awareness |
| `ci-workflows.md` | `fileMatch: **/.github/**` | CI workflow awareness |
| `kiro-compatibility.md` | `manual` | Kiro-specific usage guide |
| `project-framework.md` | `always` | Auto-generated framework conventions (optional) |

### Hooks (`.kiro/hooks/`)

| Hook | Event | Action |
|------|-------|--------|
| `review-write-ops.json` | `preToolUse` (write) | Check against Forbidden Actions |
| `test-after-task.json` | `postTaskExecution` | Run tests after task |
| `lint-on-save.json` | `fileEdited` (*.php) | Run PHPCS linter |

### Key Differences

| Feature | Cursor | Kiro |
|---------|--------|------|
| Rules | `.cursor/rules/*.mdc` | `.kiro/steering/*.md` |
| Frontmatter | `alwaysApply`, `globs` | `inclusion`, `fileMatchPattern` |
| Subagents | Direct agent files | `invokeSubAgent` + `contextFiles` |
| Hooks | `.cursor/hooks.json` (shell scripts) | `.kiro/hooks/*.json` (declarative) |
| Hook format | Bash scripts with JSON I/O | JSON with `askAgent`/`runCommand` |
| Skills | Same structure | Same structure |

See `examples/cursor/hooks-guide.md` and `examples/kiro/` for detailed walkthroughs.

## How it works

The intended flow is:

1. IDE loads repo rules and `AGENTS.md`
2. Agent checks memory and current task context
3. Risky paths handled more carefully than low-risk paths
4. Work done through constrained roles, templates, and checks
5. Stable outcomes written back into project memory

In short: **human intent sets direction, repo rules shape execution, AI agents do the work**.

## Quick Start

### Source-Based Workflow (Recommended)

This project uses a **source-based compilation workflow**. All rules, agents, skills, and hooks are maintained in `src/` and compiled to IDE-specific formats.

#### 1. Clone and Setup

```bash
git clone https://github.com/YOUR_USERNAME/cursor-harness.git
cd cursor-harness
cd tools && npm install && cd ..
```

#### 2. Compile to IDE Format

```bash
# Compile for both Cursor and Kiro
node tools/harness-compiler.js compile --source

# Or compile for specific IDE
node tools/harness-compiler.js compile --source --target=cursor
node tools/harness-compiler.js compile --source --target=kiro
```

#### 3. Open in Your IDE

Open the project in Cursor or Kiro. The compiled configurations will auto-load.

#### 4. Customize (Edit Source Files)

```bash
# Edit rules
src/rules/core-protocol.md
src/rules/php-guardrails.md

# Edit agents
src/agents/coder.md
src/agents/tester.md

# Edit hooks
src/hooks/check-forbidden-actions.yaml
src/hooks/review-write-ops.yaml

# Edit skills
src/skills/mcp-setup/SKILL.md

# After editing, recompile
node tools/harness-compiler.js compile --source
```

#### 5. Validate Configuration

```bash
# Validate source files
node tools/harness-compiler.js validate --source
```

### Compiler Tool Usage

The `tools/harness-compiler.js` tool compiles unified source files to IDE-specific formats.

#### Commands

```bash
# Compile from src/ directory
node tools/harness-compiler.js compile --source [--target=cursor|kiro|all]

# Compile from harness.yaml (alternative)
node tools/harness-compiler.js compile [--target=cursor|kiro|all]

# Validate configuration
node tools/harness-compiler.js validate --source

# Show help
node tools/harness-compiler.js help
```

#### Options

- `--source` - Use `src/` directory as source (recommended)
- `--target=cursor` - Compile only for Cursor
- `--target=kiro` - Compile only for Kiro
- `--target=all` - Compile for both (default)

#### What Gets Compiled

| Source | Cursor Output | Kiro Output |
|--------|---------------|-------------|
| `src/rules/*.md` | `.cursor/rules/*.mdc` | `.kiro/steering/*.md` |
| `src/agents/*.md` | `.cursor/agents/*.md` | `.kiro/agents/*.md` |
| `src/skills/*/SKILL.md` | `.cursor/skills/*/SKILL.md` | `.kiro/skills/*/SKILL.md` |
| `src/hooks/*.yaml` | `.cursor/hooks/*.sh` + `hooks.json` | `.kiro/hooks/*.json` |

#### Automatic Transformations

The compiler automatically handles:
- **Frontmatter translation**: `inclusion: always` ↔ `alwaysApply: true`
- **Hook format conversion**: YAML → shell scripts (Cursor) or JSON (Kiro)
- **Path reference updates**: Adjusts file paths for each IDE
- **IDE-specific features**: Adds IDE-specific metadata

See `README_COMPILER.md` for detailed compiler documentation.

## Project philosophy

This repo follows a simple idea:

**Do not try to make the agent smarter only through prompting. Make the environment better instead.**

That means:

- clearer constraints
- smaller blast radius
- reusable review logic
- persistent memory
- explicit operational limits
